# AGENTS.md — JARVIS PR Review Bot

Инструкции для AI-агентов (opencode и совместимых), работающих в этом репозитории.

## Что это за проект
**JARVIS PR Review Bot** — внутренний бот для авто-ревью Perl-PR в Bitbucket банка.
Stateless-сервис: webhook от Bitbucket → забирает diff → шлёт во внутренний Феникс
(DeepSeek V3.2 через LiteLLM) → постит inline-комментарии в PR. **Advisory-only** —
финальное решение всегда за сеньором-человеком.

## Стек
- Python 3.11 + FastAPI + Docker, без состояния (источник состояния — сам PR в Bitbucket).
- Развёрнут на банковской VM (dev), **без публичного интернета** — только внутренние API.
- Деплой: `docker build` → `docker save -o pr-review-bot-vN.tar` → перенос на VM →
  `docker load` + `docker-compose up`. Тег образа всегда `pr-review-bot`, версия — в имени .tar.

## Жёсткие границы (не нарушать)
- ❌ **Никаких внешних LLM** — только внутренний Феникс.
- ❌ **Бот не пушит/не мёрджит/не аппрувит** — только GET diff + POST comment.
  Сервисный токен: права Repo:Read + PR:Write (НЕ Repo:Write).
- ❌ **Секреты только в ENV** — никогда в коде. `.env` не коммитить (в git — только `.env.example`).
- ❌ **Стайлгайд монтируется снаружи** (`/app/styleguide.md`) — менять через файл на VM, не в образе.

## Конвенции репозитория
- Каноничная ветка — **`master`** (она же default). Веток `main` нет.
- **Коммиты — от автора `jarvis-bot <jarvis-bot@local>`**, НЕ от личной почты
  (личные данные не должны утекать в публичный репозиторий). Перед пушем проверяй,
  что не уходят секреты, `.env`, тестовые артефакты и личные имена/почты.
- **Без персональных имён** в коде, коммитах и доках — только роли (владелец, DevOps, сеньор).
- В образ (Dockerfile) попадает только `pr_review_bot.py` — тестовые фикстуры в репо не держим.
- Деплой делает владелец вручную через .tar; `git push` — не часть деплой-флоу,
  но репозиторий держим чистым.

## Ключевые поведения кода (чтобы не сломать при правках)
- `parse_bitbucket_diff` помечает строки реальным номером `[L<n>]` из поля `destination`
  Bitbucket diff-JSON; если номера нет — метку НЕ ставит (иначе ломается inline-привязка).
- Ревью идёт **пофайлово** (`MAX_DIFF_LINES` — лимит на файл, не на весь PR), не проверенные
  файлы честно перечисляются в PR.
- Дедуп комментариев на повторном прогоне — чтением существующих комментариев PR (метод
  «игнорировать», без удаления). Не вводи собственное хранилище состояния — это нарушит stateless.
- Промпт подаёт diff и стайлгайд как **данные** в границах `«DIFF»`/`«STYLEGUIDE»`; инструкции
  идут ПОСЛЕ данных и приоритетнее их (защита от prompt injection). Не разрушай этот порядок.
- Webhook слушает события `pr:opened`, `pr:from_ref_updated` (пуш коммитов — НЕ `pr:modified`!),
  `pr:modified` (метаданные).

## Ревью
Для код-ревью Perl-PR используй сабагента **`jarvis-reviewer`** (`.opencode/agent/`) —
он заточен под банковские критерии (инъекции, валидация, `open`, секреты, стайлгайд)
с приоритизацией P0/P1/skip.

## Стиль работы агента
- Прагматизм важнее перфекционизма: не навязывай полный спек-флоу на багфикс/соло-правку.
  Нитпики — пушбэк, не выдавай как блокеры.
- Меняй по одному шагу, объясняй до и после, минимизируй blast radius.
- Не выдумывай требования и не переписывай архитектуру — исполняй, не изобретай.
