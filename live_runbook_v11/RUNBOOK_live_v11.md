# Live-runbook — добивка VM-пунктов на сборке v11

Прогон **только того**, что нельзя проверить дома: живой Bitbucket + Феникс + контейнер.
Локальная часть (парсинг, SEC-003/005, REV-004, дедуп, /health) уже закрыта `verify_v11.py` — 23/23.

> ⚠️ Это сборка **v11** (фикс `[LNone]` в неё НЕ входит — он для v12). На этих фикстурах это
> не мешает: все строки тут обычные, у них есть `destination`, поэтому номера строк на v11
> отрабатывают штатно и Шаг 6 проверяется честно.

**Описание тестов — здесь, в этом файле. Внутри `.pl` пояснений НЕТ** (чтобы мета-текст не попал
в diff и не подсказал боту). Где какой баг заложен — см. таблицы ниже.

### Порядок прогона — 4 PR, сгруппированы по стайлгайду

Стайлгайд (`styleguide.md`) подменяется **всего 2 раза** — поэтому порядок такой:

| PR | Файл(ы) | Стайлгайд на VM | Тест |
|---|---|---|---|
| 1 | `04_injection_attack.pl` | рабочий (из `.bak`) | **C** — инъекция |
| 2 | `01`+`02`+`03` (один PR, 3 файла) | рабочий (тот же) | **D** — multi-file + cached_tokens |
| 3 | `05_clean.pl` | единорог | **B** — стайлгайд работает |

Сначала PR1+PR2 на **рабочем** стайлгайде → одна подмена на единорога (PR3) → одна на cp1251 (PR4)
→ восстановить. Контейнер не перезапускаем (hot-reload), но дай боту 10–30 сек на каждый PR.

## Что принести мне (по каждому тесту)

1. **Комментарии бота** — текст/скрин из PR в Bitbucket: severity, **на какой строке** висит инлайн, дословная формулировка.
   *Где взять:* вкладка PR в Bitbucket UI.
2. **Лог контейнера** по каждому прогону — `sudo docker logs pr-review-bot --tail 80`.
   Мне нужны строки-маркеры:
   - `📂 Файлов в PR #...: N — [...]`
   - `✅ Стайлгайд загружен` **или** `⚠️ Стайлгайд не в UTF-8...`
   - `📤 Отправка в Феникс: ...`
   - `📥 Ответ Феникса: finish_reason=..., prompt_tokens=..., cached_tokens=..., completion_tokens=...`  ← **это и есть §4 cached_tokens**
   - `✅ Ревью завершено. Файлов проверено K/N, ...`
   *Где взять:* SSH на VM → команда выше (можно `--tail 120`, если прогонов несколько).

Удобный минимум: для тестов A/B/C — по 1 скрину комментариев + хвост лога; для D — скрин всех трёх файлов + полный лог одного прогона (включая строку `cached_tokens`).
